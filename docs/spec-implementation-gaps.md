# 仕様と実装のギャップ一覧

作成日: 2026-02-10

## 概要

`docs/SPEC.md`、`docs/api-spec.md`、`docs/specs/*.md` の仕様書と現在の実装を比較し、**仕様通りに動いていない箇所**を優先度別に一覧化しました。

---

## 🔴 優先度: 高（セキュリティ・機能的影響大）

### H-1. 商品詳細取得時の公開状態チェックがない

**仕様**: SPEC.md:179「非公開（isPublished = false）の商品は表示しない」
**実装**: `ProductService.java:52-57` - 商品詳細API (`GET /api/item/:id`) では公開状態をチェックしていない
**問題**: 非公開商品もIDが分かれば誰でも取得できる（セキュリティ問題）
**影響**: 公開前の商品情報が漏洩する可能性
**対応**: 商品詳細取得時に `isPublished` チェックを追加するか、仕様を変更

---

### H-2. 在庫引き当て処理が未実装

**仕様**: SPEC.md:406-408「今後の実装予定: 注文時の在庫引き当て」
**実装**: 注文作成時に在庫チェックは行うが、在庫数（stock）を減らす処理がない
**問題**:
- 在庫数を超える注文が理論上可能（複数ユーザーが同時注文した場合）
- 在庫管理が正確に機能しない
- オーバーセリング（過剰販売）のリスク

**影響**: 実運用では致命的な問題
**対応**:
- Phase 1スコープ外として明確化し、制約事項に明記
- または在庫引き当て機能を実装（トランザクション制御・排他制御を含む）

**参照**:
- `inventory.md:180-200`（在庫引当の詳細説明）
- `gap-analysis.md:133-136`

---

### H-3. 注文状態遷移機能が未実装

**仕様**: SPEC.md:284-338「注文の状態遷移」（PENDING → CONFIRMED → SHIPPED → DELIVERED）
**実装**: 注文は常に `PENDING` 状態のまま。状態遷移のAPIエンドポイントが存在しない
**問題**:
- 注文ステータスが変更できない
- 発送・配達完了の管理ができない
- 仕様書に詳細な状態遷移図があるのに未実装

**影響**: 注文管理機能が不完全
**対応**:
- Phase 1スコープ外として明確化
- または状態遷移API（PATCH `/api/order/:id/status`）を実装

**参照**:
- `order.md:294-354`（状態遷移の詳細説明）
- `gap-analysis.md:123-126`

---

## 🟡 優先度: 中（UX・データ整合性）

### M-1. 配送料・手数料が常に¥0（仕様が曖昧）

**仕様**: SPEC.md:104-105「配送料、手数料」の表示が記載されている
**実装**: 配送料・手数料は常に¥0固定。計算ロジックが存在しない
**問題**:
- 配送料・手数料の計算方法が仕様に記載されていない
- 0円固定なのか、将来実装予定なのか不明

**影響**: ビジネスロジックの不完全性
**対応**: 仕様書に「現在は配送料・手数料ともに¥0固定。将来的に実装予定」と明記

**参照**:
- `pricing.md:150-194`
- `order.md:461-479`
- `gap-analysis.md:118-121, 167-171`

---

### M-2. 在庫不足時の詳細エラーレスポンスが未実装

**仕様**: api-spec.md:437-453「OUT_OF_STOCK エラー時に `details` 配列で商品ごとの在庫状況を返す」
**実装**: 単純なエラーメッセージのみ返す（どの商品が不足しているか不明）
**問題**:
- 複数商品がカートにある場合、どの商品が在庫不足か分からない
- ユーザーが対処しづらい

**影響**: ユーザー体験の低下
**対応**:
- `details` 配列を実装して在庫不足商品を明示
- または仕様から削除

**API仕様の例**:
```json
{
  "success": false,
  "error": {
    "code": "OUT_OF_STOCK",
    "message": "在庫が不足している商品があります",
    "details": [
      {
        "productId": 1,
        "productName": "ワイヤレスイヤホン",
        "requestedQuantity": 20,
        "availableStock": 12
      }
    ]
  }
}
```

**参照**:
- `inventory.md:322-326`
- `gap-analysis.md:143-146, 232-235`

---

### M-3. 非公開商品がカートに残る問題

**仕様**: SPEC.md:179「非公開の商品は表示しない」
**実装**: カートに入れた後、商品が非公開になってもカート内に残り続ける
**問題**:
- 仕様が曖昧（カート内の扱いについて記載なし）
- 非公開商品が注文できてしまう可能性

**影響**: ビジネスロジックの不整合
**対応**:
- 仕様を明確化（「非公開になってもカートに残る」または「自動削除する」）
- 自動削除する場合は実装追加

**参照**:
- `admin.md:275-306`（シナリオ 3-2）
- `gap-analysis.md:197-201`

---

### M-4. 注文キャンセル機能が未実装

**仕様**: SPEC.md:329-332「PENDING/CONFIRMED → CANCELLED への遷移」
**実装**: キャンセル機能のAPIエンドポイントも画面も存在しない
**問題**:
- ユーザーが注文をキャンセルできない
- 管理者も注文をキャンセルできない

**影響**: 顧客対応の柔軟性欠如
**対応**: Phase 1スコープ外として明確化

**参照**:
- `inventory.md:222-239`
- `order.md:340-353`
- `gap-analysis.md:128-131`

---

### M-5. カート内商品の数量上限が不明

**仕様**: SPEC.md:86-89「数量変更機能」の記載のみ
**実装**:
- バックエンド: 在庫数までという暗黙のルール
- フロントエンド: 入力欄に `max` 属性なし

**問題**:
- 1商品あたりの最大購入数量が不明
- 大量購入時のオーバーフロー対策なし

**影響**: データ整合性のリスク
**対応**:
- 仕様に「カート内の数量は在庫数を超えられない」と明記
- フロントエンドで入力制限を設ける（例: max={product.stock}）

**参照**:
- `inventory.md:282-286`
- `pricing.md:422-433`
- `gap-analysis.md:155-159`

---

### M-6. エラーコード一覧が不完全

**仕様**: api-spec.md:575-586「エラーコード一覧」
**実装**: 仕様に記載のないエラーコードが使用されている
**差異**:
- `CART_NOT_FOUND` - バックエンドで使用（CartService.java）
- `NETWORK_ERROR` - フロントエンド独自（api.ts）

**影響**: API仕様書の不完全性
**対応**: エラーコード一覧を最新化

**参照**: `gap-analysis.md:249-254`

## 🟢 優先度: 低（実装詳細・将来の拡張）

### L-1. 商品一覧のページネーション機能（フロントエンド未実装）

**仕様**: api-spec.md:42-46「ページングパラメータ（page, limit）」
**実装**:
- バックエンド: ページングをサポート
- フロントエンド: 全件取得のみ（ページング機能なし）

**問題**: 商品数が多い場合にパフォーマンス問題が発生する可能性
**影響**: 現状は商品数が少ないため問題ないが、将来的に必要
**対応**: Phase 2以降で実装

**参照**: `gap-analysis.md:138-141`

---

### L-2. おすすめ商品の選定ロジックが単純

**仕様**: SPEC.md:17-30「AIがおすすめする商品」
**実装**: 単に最初の3件を表示しているだけ
**問題**: 「AI」レコメンデーションではない
**影響**: プロトタイプ段階では許容範囲
**対応**: 仕様に「将来的にAIレコメンデーション実装予定」と明記

**参照**: `gap-analysis.md:148-151`

---

### L-3. セッションIDの有効期限が不明

**仕様**: api-spec.md:172-174「X-Session-Id ヘッダー」
**実装**:
- フロントエンド: localStorage に永続保存
- バックエンド: 有効期限なし

**問題**: セッションIDが永久に有効（セキュリティ上の懸念）
**影響**: 現状は問題ないが、本番環境では不適切
**対応**: 仕様に「現在はセッションIDに有効期限なし。将来的にセッション管理機能を実装予定」と明記

**参照**: `gap-analysis.md:185-189`

---

### L-4. 注文番号の上限（999件/日）が不明

**仕様**: SPEC.md:224「ORD-YYYYMMDD-XXX 形式」
**実装**: 1000件目以降も生成されるが、フォーマットが崩れる（4桁になる）
**問題**: 1日あたりの注文上限が不明
**影響**: 通常運用では問題ないが、大規模サイトでは考慮が必要
**対応**: 仕様に「1日あたりの注文上限は999件」または「999件を超えた場合の挙動」を明記

**参照**: `gap-analysis.md:179-183`

---

### L-5. 商品画像がプレースホルダーのみ

**仕様**: SPEC.md:349「image: string // 商品画像URL」
**実装**: プレースホルダーサービス（placehold.co）を使用
**問題**: 本番環境では使用不可
**影響**: プロトタイプ段階では問題なし
**対応**: 仕様に「現在はプレースホルダー画像を使用。将来的には画像アップロード機能を実装予定」と明記

**参照**: `gap-analysis.md:173-177`

---

### L-6. 管理画面のアクセス制限なし

**仕様**: SPEC.md:143-162「管理画面」
**実装**: 誰でもアクセス可能（URLを知っていれば）
**問題**: 認証・認可機能がない
**影響**: プロトタイプ段階では許容範囲
**対応**: SPEC.md の制約事項に既に記載あり（「現在は認証・認可なし。Phase 2で実装予定」）

**参照**: `gap-analysis.md:209-213`

---

### L-7. 価格計算のオーバーフロー対策なし

**仕様**: なし
**実装**: `Cart.java:78-82` で `int` 型を使用
**問題**:
- 高額商品 × 大量数量でオーバーフロー発生の可能性
- 例: 100万円 × 2,200個 = 22億円（Integer最大値超過）

**影響**: 通常運用では問題ないが、極端なケースで不具合の可能性
**対応**:
- カート内の商品数量に上限を設定（例: 99個まで）
- または `Long` 型への変更を検討

**参照**: `pricing.md:399-433`

## 推奨対応アクション

### 短期対応（Phase 1内で対応）

1. **H-1**: 商品詳細取得時の公開状態チェックを追加（セキュリティ修正）
2. **M-6**: エラーコード一覧をapi-spec.mdに追記（ドキュメント修正）
3. 仕様書への明記（曖昧な仕様の明確化）:
   - **M-1**: 配送料・手数料は¥0固定
   - **L-2**: おすすめ商品は暫定的に先頭3件
   - **L-3**: セッションIDは有効期限なし
   - **L-4**: 注文番号の1日上限は999件
   - **L-5**: 商品画像はプレースホルダー使用

### 中期対応（Phase 2で検討）

1. **H-2**: 在庫引き当て処理の実装
2. **H-3**: 注文状態遷移機能の実装
3. **M-2**: 在庫不足時の詳細エラーレスポンス実装
4. **M-3**: 非公開商品のカート自動削除機能
5. **M-4**: 注文キャンセル機能の実装
6. **L-1**: フロントエンドのページネーション機能

### 長期対応（Phase 3以降）

1. **L-6**: 認証・認可機能（管理画面のアクセス制限）
2. AIレコメンデーション機能
3. 画像アップロード機能
4. セッション管理機能（有効期限・更新）
